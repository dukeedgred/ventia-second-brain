High-level domains for the Transport data asset

Red text shows what changed or what needs to be challenged based on table documentation and Databricks source-table validation.

1. Contract & Network context (the spine everything hangs off)

  • Contract (WRU, SRAP-C, TSRC, RAMC Gen2, Brisbane Airport, Port of Brisbane, SHT/WHT, VentureSmart, etc. SRAPC, VRMC, NEL, City Harbour as source-validated examples), business unit, client, operational lifecycle/status where present lifecycle stage (Bid -> Mobilise -> Plan -> Inspect -> Renew -> Report) as a source-table structure, contract type (roads vs tunnel), commercial model (useful interview context, but not source-table validated as the driver of item-level data).
  • Network / route / segment / linear referencing should be modelled through the available spatial fields: asset hierarchy, WKT/spatial information, chainage and contract-specific route/asset structures where populated.

2. Asset register & hierarchy (your "hierarchy of assets and their properties")

  • Parent asset: parent asset / network asset where populated Road (or Tunnel) as the universal parent.
  • Child asset categories: pavements, drainage (pits, lines), vehicle barriers/guardrails, signage, bridges, culverts, lighting, jet fans (tunnels), etc. Keep these as contract-specific examples, not a guaranteed global list.
  • Per-asset attributes: ID, type, sub-type, location/geometry, dimensions, material, install date, status, parent asset, chainage/WKT where populated, source/provenance where captured. Ownership (ours vs disputed vs client's) and AI-detected as asset provenance were not validated from the source tables.
  • Asset condition: condition exists but is patchy by contract. Keep condition definitions contract-scoped; do not assume one global 1-5 scale works everywhere without mapping.

3. Inspections (separate from defects)

  • Scheduled inspection runs are source-backed where scheduled dates are populated, but coverage varies by contract.
  • Inspection record: who, when, what asset/route was covered, method (manual / field inspection / Asset Vision inspection forms / tunnel maintenance or event forms where present), evidence (imagery, photos, video/GPS trail where populated). RetinaVision, AID camera inspections and dash-cam capture intervals were not validated from source tables.
  • Link to assets observed and condition assessments made where those fields are populated; some inspections are job-linked rather than asset-linked.

4. Defects & Hazards (one model, severity-flagged)

  • Raised from: inspection, public report, incidental observation where supported by source forms or interview evidence. AID alert as a validated source-table trigger.
  • Attributes: asset link, defect type / hazard-defect code, severity/hazard level where populated, photo evidence, raised by, raised at, location/WKT.
  • SLA / response time: due date and compliance fields are source-backed. The exact auto-calculation rule based on severity + asset category is not proven from the source tables alone and should be contract-scoped.

5. Jobs / Work orders (the response side)

  • Triggered by defect/hazard OR scheduled maintenance with capital work links where present. Capital works proposal as a universal direct trigger was not validated.
  • Crew/resource assignment, scheduled date, response SLA/due date, completion timestamp, before/after/during photos, completion status.
  • Routine maintenance vs Capital Works distinction is useful, but the rule "anything routine can't fix -> capital works" should be treated as interview-backed.

6. Sensing & telemetry (your "sensing side")

  • Source forms contain camera/CCTV, GPS-like location fields, tunnel equipment alarm/check fields, OMCS references, fan-related maintenance fields and AI Defect ID.
  • Asset Vision dash cam every 5-10m, RetinaVision every 3m, AID cameras for tunnel incident detection, and fleet telematics were not validated from the source tables.
  • Sensor -> event -> incident/work order flow should be written as interview-backed architecture, not as confirmed source-table lineage.

7. KPIs & Reporting

  • Contract KPI definitions vary by contract and should be modelled as contract-scoped. Source tables validate measurement inputs, not necessarily the full KPI rule library.
  • KPI measurements: inspections scheduled vs completed, jobs raised vs closed within SLA/compliance, asset condition distribution where condition is populated, pavement performance fields where present in contract-specific forms.
  • Monthly contract report outputs remain interview/process evidence. The 50+ page Word report claim was not validated from source tables.

8. Incidents (tunnel-specific, distinct from defects)

  • Tunnel event/incident handling appears in source forms and OMCS-related wording, but the source-table run did not validate a complete OMCS incident-management table/feed.
  • Keep incidents conceptually separate from defects/work orders, but label the OMCS incident-management detail as interview-backed unless a separate OMCS source is provided.

A few things I'd flag before locking this in

  • Contract is the partitioning key. This is validated. The source data shows shared core tables plus contract-specific form/module extensions. Do not claim mandatory-field standardisation solely from the source tables; keep Rui's point as interview evidence.
  • Roads vs Tunnels diverge meaningfully in the sensing layer. This is directionally right, but only partially source-backed. Always-on AID + OMCS alarms as a fully validated source-table flow is too strong.
  • As-built / handover data is its own beast. Handover/practical completion and as-built mismatch references are source-backed. PDFs + asset registers + GeoJSON at handover were not validated from the source tables.
  • You're missing one I'd add: Capital Works / Forward Programme. Keep this. It is source-backed by capitalwork/vcapitalwork tables and should be a distinct domain alongside jobs. dTIMS fields are source-backed in forms, but not a confirmed direct dTIMS system feed.
