High-Level Domains for the Transport Data Asset

Correction basis: updated against current Transport table documentation. Green text is added or corrected. Red text is removed or downgraded because the table docs do not validate it.

1. Contract & Network Context

(the spine everything hangs off) (one of the core partitioning and governance contexts; not the only spine)

  • Contract: WRU, SRAPC, VRMC, NEL, City Harbour, etc. WRU, SRAPC, NEL, RAMC, TSRC, SHT/WHT, VentureSmart, Auckland West, and other contract/schema contexts where the source-system or table documentation supports the mapping.
  • Business unit, client, contract type: keep these as required reference attributes, but do not assume they are all present in current tables.
  • Lifecycle stage: Bid → Mobilise → Plan → Inspect → Renew → Report useful operating framework, but not currently validated as a table-backed domain.
  • Commercial model: lump sum drawdown vs scheduled rates — Rui called this out as a key driver of why item-level data exists or doesn't commercial model should be captured if available, but current table docs validate SAP/WBS/job-costing links rather than this specific commercial-model explanation.
  • Network / route / segment / linear referencing: confirmed as a separate spatial spine. Use route, section, road ID, asset ID, direction, chainage, WKT, latitude/longitude, and reference-point fields where available.

2. Asset Register & Hierarchy

This domain is validated, but it is not yet proven as a complete enterprise asset master across all contracts.

  • Parent asset: Road, tunnel, bridge, segment, counter location, or other parent asset depending on contract/source.
  • Child asset categories: pavements, drainage, pits, lines, vehicle barriers/guardrails, signage, bridges, culverts, lighting, jet fans and tunnel/building assets where a contract source actually documents them.
  • Per-asset attributes: ID, code, name, type, asset type, parent asset, direction, chainage, location/geometry, dimensions, material, construction/install date, status/ stage, disposal date, criticality, condition, risk, classification, notes, alert notes, WKT/lat-long where available.
  • Ownership and provenance: ours vs disputed vs client's, source/provenance as-built handover, field-validated, AI-detected model this as a required capability, but current table docs only partially support it through fields such as information source, data confidence, verification notes, audit/export metadata, upload-source notes, and asset status/stage. Do not claim a mature as-built/AI-detected provenance model yet.
  • Asset condition: scored 1–5 very good → very poor, with definitions agreed in contract condition fields exist, but scales vary by source: Asset Vision condition/risk fields, NEL condition rating, TSRC bridge condition/grade, WRU counter condition/risk, RAMC roughness/rutting/ride quality classes. Treat condition as contract-scoped until a common scale is formally mapped.

3. Inspections

Keep separate from defects/jobs. This is table-supported.

  • Scheduled inspection runs: validated through inspection tables and SHT inspection compliance logic with scheduled date, due date, completed date, assigned user, route/ type, and compliance flag.
  • Inspection record: who, when, route/asset covered, inspection type, status, scheduled date, completed date, completed user, comments, WKT/chainage where available.
  • Method: drive-by with Asset Vision dash cam / RetinaVision / manual / AID camera in tunnels do not state these as table-validated methods unless backed by source-specific fields. Current table docs validate inspection records, not every capture method.
  • Evidence: photos/URLs and form/module fields are table-supported; video/GPS trail should be treated as a requirement or interview claim unless separately evidenced.
  • Link to assets observed and condition assessments: supported via asset IDs, asset codes, inspection IDs, job IDs, capital work IDs, and WKT/chainage fields.

4. Defects & Hazards

(one model, severity-flagged) (shared operational issue/job pattern where fields support it, with hazards kept explicit because they carry safety semantics)

  • Raised from: inspection, AID alert (tunnels), public report, incidental observation CRM/public report fields and operational modules where documented. AID alert should not be claimed from current table docs.
  • Attributes: asset link, hazard code, hazard/defect code, activity type/category/code, priority, created date, due date, completed date, completed status, compliance flag, made-safe flag/date, comments, photo evidence, location/WKT/chainage.
  • SLA / response time: Auto-calculated SLA / response time based on severity + asset category table docs validate due dates, compliance flags, compliance calculation dates, and contract-specific compliance logic. They do not prove a universal severity + asset-category SLA engine.

5. Jobs / Work Orders

This domain is strongly table-supported.

  • Triggered by: defect/hazard, inspection, scheduled maintenance, CRM/public report, OR capital works proposal or linked capital works where the data actually connects the records.
  • Core attributes: job/work-order ID, asset ID/code/name, contract, region, activity/intervention fields, priority, assigned user, scheduled start/end, due date, completion timestamp, completion status, compliance, workflow status, comments, photos, quantity, cost estimates, WKT/chainage.
  • Crew/resource/cost capture: add resources, planned resource items, timesheet items, labour/resource cost, SAP work order, WBS, and job-costing fields.
  • Routine maintenance vs capital works: keep the distinction. Capital works has separate Asset Vision tables and should not be collapsed into ordinary jobs.

6. Capital Works / Forward Programme

Added as its own domain because the table docs support separate capital work and capital work task structures.

  • Capital work ID, name, description, type, asset type, asset ID/code/name, contract, section, direction, chainage, planned start/finish, actual start/finish, supervisor, area, spatial info.
  • Capital work task ID, capital work ID, task name/description, planned/actual dates, estimated/actual quantity, estimated/actual cost, unit, comments, linked job ID.
  • pavement modelling in dTIMS, multi-year capex proposals to client keep as interview/future-state context unless table evidence is added. Current table docs prove capital works records, not the full dTIMS/ proposal workflow.

7. Evidence, Forms, Photos & Workflow Extensions

Added because this is not a side detail; Asset Vision tables make it part of the operational model.

  • Photos: source table, source record ID, URL, thumbnail URL, stage, created date/user.
  • Form fields: source table, source record ID, field name, value, data type, sort order.
  • Workflow status: source table, source record ID, workflow code/name, status date, comment, reason.
  • Modules/custom module items: contract-specific registers and operational forms.

8. Resources, Timesheets & Job Costing

Added because the current table docs validate resource, timesheet, utilisation, and job-costing data.

  • Resource code/name/type, company rate, hours, minutes, quantity, multiplier, cost, start/end dates.
  • Timesheet source table/source ID, timesheet type, created user/date, resource cost.
  • All-contract job-costing context: contract, Asset Vision database, activity, record ID, job status, asset type/code/name, WBS number, SAP work order, SAP record existence.

9. SAP / WBS / Finance Context

Added as a first-class domain. It is table-supported and missing from the original structure.

  • SAP work order / cost object, object type, object description, company code, cost centre, currency, object status, WBS ID, profit centre, routing number, operation activity/text.
  • WBS budget and forecast: fiscal year, version, cost element, debit/credit indicator, month amounts, WBS element/description.
  • Vendor and purchasing group data where needed for procurement/commercial analysis.

10. Sensing, Traffic, Weather & Operational Context

6. Sensing & telemetry (your "sensing side") Renamed and narrowed to what the table documentation can actually support.

  • Imagery feeds: Asset Vision dash cam every 5–10m, RetinaVision every 3m, AI defect detection, AID cameras not table-validated in the current documentation. Treat as interview/future-state inputs unless specific source tables are documented.
  • Operational telemetry: OMCS alarms, vehicle stopped, jet fan degraded, plant/equipment, GPS/fleet telematics, vehicle locations not validated by current table docs as a complete telemetry layer.
  • Table-supported context: traffic counters, traffic volumes, traffic closures, weather observations/forecast, road network centreline/reference data, closure durations, lane affected, direction, section, chainage.
  • Flow: Sensor → event → incident record OR work order is the flow Huy described keep only as a hypothesis to validate. Current tables show modules/forms/incidents/closures/jobs, but not a proven universal sensor-to-event automation flow.

11. KPIs & Reporting

Keep, but make it explicitly contract-specific and source-quality dependent.

  • Contract KPI definitions: validated by contract-specific KPI/compliance tables and views. Do not create a single global KPI schema without mapping.
  • KPI measurements: inspections scheduled vs completed, compliance pass/fail, jobs raised/closed, due/completed dates, backlog changes, asset condition distribution, pavement roughness/rutting/ride-quality classes, work order target/actual dates.
  • Monthly report outputs: 50+ page Word docs today may be true from interviews, but table docs validate reporting tables and snapshots rather than the Word-document consumption layer.
  • Data maturity: include upload-based tables, truncated/undocumented objects, decommissioned all-contract jobs caveat, and contract-specific readiness scores.

12. Incidents & Closures

8. Incidents (tunnel-specific, distinct from defects) Incidents and closures should be separate from ordinary defects/jobs where incident data exists; do not claim they are only tunnel-specific from table docs alone.

  • Incident fields: incident ID, incident type, status, status date, section, incident location, traffic direction, severity description, detection method, detection reference.
  • Closure fields: closure type, chainage, traffic flow stopped/resumed, duration, lane affected, section, TGS number, audit flag, AED external ID, AV incident ID, completion code.
  • Non-asset-affecting events captured in OMCS incident management — worth keeping separate from work orders per Huy current table docs support incident/closure separation, but not the OMCS/Huy-specific statement.

Things to Flag Before Locking This In

  • Contract is the partitioning key. Correct, but not sufficient. The model also needs asset hierarchy, route/section/chainage/WKT, job/work-order IDs, inspection IDs, and SAP/WBS cost objects as structural keys.
  • Asset Vision schemas are standard across mandatory fields, with contract-specific forms. Supported by repeated Asset Vision source tables plus formfield/module/custom module patterns. This is a strong modelling principle.
  • Roads vs tunnels diverge in sensing. Roads = inspector-driven imagery. Tunnels = always-on AID + OMCS alarms. Likely interview context, but current table docs only validate some road/tunnel contract differences, inspection compliance, traffic/closure/ incident data, weather, and Asset Vision modules/forms. Do not hard-code AID/OMCS without source evidence.
  • As-built / handover data is its own beast. Huy described receiving PDFs + asset registers + geojson at handover... Keep provenance as a required domain, but mark PDFs/GeoJSON/handover translation as interview evidence not table-doc evidence.
  • Capital Works / Forward Programme. Keep and promote to its own domain. Current tables support capital work and capital work task records. dTIMS and multi-year proposal workflow still need table/source confirmation.
